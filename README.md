

# miniMoE
> 手搓 1.5B MoE 模型，融合前沿架构优化，有限资源 + 大数据集下完成训推，深入掌握MoE  
> Build 1.5B MoE from scratch with advanced architectural optimizations, complete training & inference under limited resources and large datasets, deeply master MoE core technology

## 1. 项目介绍 / Introduce

以 DeepSeek、Qwen、Gemini、GPT-4o 为代表的业界顶尖大语言模型，均采用 MoE 混合专家架构，凭借稀疏计算、动态激活的核心优势，成为当下大模型高性能扩容与低成本高效训练的主流核心技术路线。

miniMoE 项目基于 PyTorch 从零自研搭建轻量化混合专家模型（参考前置项目：TransDemo、miniGPT），融合多项前沿模型结构优化与底层性能增强技术；在有限显存资源下极致压榨硬件潜力、放大模型参数量与能力上限，依托大规模开源低质数据集完成迭代训练优化，最终落地实现具备基础流畅交互能力的原生对话生成模型。

本项目完整覆盖先进 MoE 架构设计、显存优化、大数据处理、训练调优与推理全流程，聚焦拆解 MoE 核心结构、前沿创新技术与低显存高效训练等关键底层原理，理论结合实战，深入掌握 MoE 大模型底层原理与工程能力。


## 2.  效果 / Effect Demo

> 说明：  
> - 受限于训练资源，实际演示模型为裁剪到~0.5B的模型
> - 举例的输入均为在测试数据集中随机选取，且经过Double Check其未在训练集中出现过  
> - 120万QA指原始公开数据集，含商业/理财、电脑/网络、游戏、娱乐等各类别QA，而5.75万同类QA则为在这之中删选的仅电脑/网络类，且长短适合本模型的QA
> - 演示Case均采用连续5次Top-K(K=5，温度=1.2)采样，+ 最后进行一次贪心算法对比
> - 目前原始数据集质量极差，且个人暂不具备时间/精力准备或清洗大规模高质量数据集，项目主要做原理验证，而非准确回答，详细分析请参考章节4&5&6

## 3. 训练 / Training

#### 数据集处理：
- 强化数据集的离线处理，使训练过程中的数据加载逻辑最简化，且适配大数据集加载场景（内存无法存放全量数据集场景）
- 离线：将new2016zh（\~8.3G文件）处理为22亿Token（22个chunk文件，单个文件\~190MB，采用Array做INT16压缩存储），均拼接为长文本（Token ID + EOS ID），单个文件用pickle.dump导出，读取时直接load文件得到直接可用的连续的Token id List
- 在线：自定义PklDataset(Dataset)，预定义change_file_count=2000，当__getitem__取文件次数每达到change_file_count时，切换chunk文件；而change_file_count次数内，则在当前读取的chunk内，随机位置截取并返回BLOCK_SIZE长度的数据；同时，将数据集__len__采用预定义的limit_len = 100000000，使数据集支持近乎无限提供训练数据
- 离线：对预训练数据集（new2016zh）中的原始文本内容，做‘仿BPE’处理，先做单个字符（BPE用UTF-8字节码）做频率统计，最终得到不重复文字及标点等2万+个，然后分析高频词，截取了前3500个(之外的直接<UNK>)纳入词表作为元数据，之后再遍历分析原始预训练数据集，进行2轮高频连词组合，形成2-gram，3-gram，4-gram的高频词组，补充纳入词表，最后抽样检索原始数据集中连续出现<UNK>或<UNK>占比过高的句子，手动补充约20~30个特殊字符，最终形成VOCAB_SIZE=5553：

- 在线：QA数据量不大，直接组装后装入TensorDataset，然后送入DataLoader

#### 训练相关信息：
- 采用动态Batch Size，前期使用32快速探索优化，后逐步提升至64/128，最终微调阶段提升至256（显存不足以支撑更大）稳定效果
- BLOCK为128
- 交替使用AdamW和Adafactor优化器：lr=5e-5, betas=(0.9, 0.95)，Adafactor则使用自动学习率
- 采用分组weight_decay：0.03 & 0.0、0.01 & 0.0
- 采用组合调度器：LinearLR（做预热）+ CosineAnnealingLR（余弦退火）+ ConstantLR （最后观察）
- 采用混合精度：autocast（FP16）+ GradScaler
- 采用nn.CrossEntropyLoss和自定义软标签交叉熵损失交替loss，smoothing=0.1
- 使用PYTORCH_ALLOC_CONF优化显存（显存碎片问题），按需替换backend为native或cudaMallocAsync，结合预占显存可最大化利用
- 训练环境：个人PC，Win10系统，i9-9900K + 16G RAM + RTX 3090(24G)

## 5. 实现介绍 / Implementation Introduction
>  参考前置项目为：TransDemo、MiniGPT，具体实现请结合代码

#### 1.  Fine-grained Experts：
- 引入DeepSeekMoE的专家结构（2401.06066v1），在MoE层分配Share/Rounte专家，分配数量参考上述章节模型配置 
- 专家结构类似FFN，采用D_MODEL -> d_ff -> GELU -> D_MODEL，但将d_ff降维到 d_ff = D_MODEL，同时去除bias，且不采用dropout

#### 2.  MoE Layer：
- 整体结构类似GPT Layer，采用Pre-LN，自回归注意力
- LayNorm升级为RMSNorm_AMP（基于nn.RMSNorm封装，解决amp下精度不匹配Warning问题）
- 在多头注意力中引入QK Norm（也采用RMSNorm_AMP）
- 在多头注意力中引入RoPE位置编码，替代原固定/可学习位置编码
- 原自写点积注意力函数替换为框架实现，并启用内置因果掩码：F.scaled_dot_product_attention
- 在多头注意力之后，引入门控层（qwen风格），再拼接多头结果
- 专家门控：共享专家计算全部有效Token，路由专家逐Token按激活数量选Top-K，并记录全专家激活结果作为返回
- (1.5B时)通过torch.utils.checkpoint封装专家层，GPT的FFN层，节省中间显存

#### 3.  MiniMoE：
- 主体结构为前2层GPT+后N层MoE
- 输入仅词表Embedding，去除原位置编码
- 输出先接Norm（也采用RMSNorm_AMP），在通过FC到词表输出
- 接收到MoE Layer的激活专家信息，汇总后继续向外传递
- (1.5B时)输出头LM Head对齐复用Embedding权重，节省参数

#### 4.  训练：
- 预训练数据集支持无限循环取数据，直接单epoch循环，按输入steps数训练；微调数据集，根据输入steps数训练，但统计epoch计数
- 对MoE层返回的激活专家信息，计算expert-level balance loss，并加权到训练loss中
- 对单个step计算耗时，其中step开始/结束&next(loader_iter)采用CPU计时，而H2D数据、前向、Loss计算、反向、优化器，则采用torch.cuda.Event计时，最终可统计整个Step计时、每个步骤耗时，以及GPU异步利用率
- 完成每次step后，执行torch.cuda.synchronize()同步，并清理过程数据，主动释放内存/显存占用（同步对效率有影响但极小，体现在GPU异步任务与CPU任务的掩盖，但可有效避免显存OOM及显存碎片问题）
- 引入了torch.compile、tf32设置、mem_efficient_sdp\flash_sdp\math_sdp，但由于开发环境限制，未达成预期
- Dashboard改用滑动窗口监控Loss，仅查看最新N个Step的loss

#### 5.  推理：
- 重点做文本问答效果验证，聚焦Top K推理策略，结合对比贪心算法，复用前置miniGPT项目的贪心&Top K算法，省略Top-P算法


## 6. 过程总结 / Conclusion

- 使用Adafactor优化器可节省大量显存（AdamW占用2倍模型权重大小显存），但需考虑对收敛效率的影响
- 使用3090 24G显存下，基于大量显存优化技术后（checkpoint、synchronize、AMP、梯度累积等），可勉强达成1.5B模型训练（预留\~2GB显存Win系统使用，还有一些余量，但未进一步探索极限），但由于训练效率仅\~3s/step（前期尝试训练2万步，loss由8.6下降到7.2），结合计算需预训练的数据量后，预期时间过长，故先裁剪完成实验，1.5B留到后继探索
- 使用余弦退火学习率时，可观察到学习率对训练收敛的影响起到关键作用，可考虑基于探索的最佳学习率加速训练
